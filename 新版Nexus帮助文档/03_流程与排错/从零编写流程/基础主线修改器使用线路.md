# 基础主线修改器使用线路

这条主线用于先建立一条普通用户最容易理解、也最容易实际跑起来的 NeXus 使用线路。  
它的目标很明确：先让粒子产生出来，再让粒子运动起来，最后再决定结果要显示成液体表面、拖尾线条，还是继续保持粒子点。

如果后面要从零开始重写单个修改器帮助文档，建议优先从 `nx 发射器（nxEmitter）` 开始。大多数主线都会从它起步，后面的 `nx 缓存（nxCache）`、`nx 流体（nxFluids）`、`nx 网格化（nxMesher）`、`nx 拖尾（nxTrail）` 都是在它后面接出不同分支。

## 最小可工作主线

### 1. `nx 发射器（nxEmitter）`

这是主线起点。它负责创建粒子，并决定粒子从哪里出现、什么时候出现、初始速度朝哪里走。

做完这一步后，你至少应该能在视口里看到粒子被创建出来。  
如果这一步还没有结果，后面的力、流体、网格化都不会开始工作。

适合先在这里确认几件事：

- 粒子有没有真的出生
- 发射来源是不是你想要的网格、体积、边、点或样条
- 出生时间和出生数量是不是大致正确
- 初始方向和初始速度是不是朝着你预期的方向

### 2. `nx 缓存（nxCache）`

当主线已经能产生粒子后，通常就该尽早接上 `nx 缓存（nxCache）`。  
它负责把模拟结果写入缓存并回放，这样你后面继续加力、加流体、加显示输出时，观察会更稳定，也更方便反复检查。

普通用户最容易遇到的问题，是一边调参数一边来回播放时间线，结果分不清自己看到的是实时重算，还是之前留下的缓存结果。  
把缓存放进主线后，这条线路会更清楚：

- 前面负责生成和求解
- 这里负责记录和回放
- 后面负责把结果显示出来

### 3. `nx 碰撞体（nxCollider）`

只要场景里有“粒子要撞到什么东西”这个需求，这一步就应该尽早加入。  
它负责让粒子和网格对象或曲线对象产生碰撞关系。

这一步加得越早，后面看重力、风力、流体、拖尾时，结果越接近最终使用状态。  
如果你先把运动和显示都调了很多，最后才加入碰撞体，前面的判断通常又要重看一遍。

### 4. `nx 重力（nxGravity）` 或 `nx 风力（nxWind）` 或 `nx 湍流（nxTurbulence）`

这一步负责把“会出生的粒子”变成“会运动的粒子”。

常见起步方式可以这样理解：

- 想先看到最直接的下落效果，用 `nx 重力（nxGravity）`
- 想先看到单方向推动，用 `nx 风力（nxWind）`
- 想先看到不规则扰动，用 `nx 湍流（nxTurbulence）`

这三个修改器都很适合放在主线前段，因为它们会直接改变你对发射是否正常的判断。  
同样的发射器参数，在没有力和已有力的情况下，看起来会像两个完全不同的结果。

## 常见主线分支

当前面的主线已经能稳定地产生、缓存、碰撞、运动之后，后面通常会分成下面几条常见分支。

### 分支 A：粒子结果分支

如果你的目标就是保留粒子点、粒子方向或粒子属性本身，那么主线走到前面的运动层就够用了。  
这条分支适合继续补充粒子层控制，例如分组、颜色、缩放、速度、旋转这类修改器。

这也是最适合写第一批帮助文档的主线，因为用户能最快把“参数变化”和“视口结果变化”直接对应起来。

### 分支 B：液体结果分支

如果你的目标是液体、浆体、流体块面、翻涌体积这一类结果，通常会从前面的主线转入：

`nx 发射器（nxEmitter）`  
`-> nx 缓存（nxCache）`  
`-> nx 碰撞体（nxCollider，可选但常见）`  
`-> nx 流体（nxFluids）`  
`-> nx 网格化（nxMesher）`

这里每一步的职责可以这样理解：

- `nx 发射器（nxEmitter）` 负责把粒子放出来
- `nx 流体（nxFluids）` 负责把这些粒子按流体求解器来运动
- `nx 网格化（nxMesher）` 负责把粒子结果变成可见的表面网格

做完 `nx 流体（nxFluids）` 后，你看到的重点通常还是“流得对不对”。  
做完 `nx 网格化（nxMesher）` 后，你看到的重点会变成“表面连不连、厚不厚、细节够不够”。

### 分支 C：线条与轨迹结果分支

如果你的目标是轨迹线、枝条感、路径感、连续线段结果，主线通常会转入：

`nx 发射器（nxEmitter）`  
`-> nx 缓存（nxCache）`  
`-> nx 重力（nxGravity）` / `nx 风力（nxWind）` / `nx 湍流（nxTurbulence）`  
`-> nx 拖尾（nxTrail）`

`nx 拖尾（nxTrail）` 会把粒子运动过程整理成线条或轨迹结构。  
所以在接入它之前，前面的粒子运动最好已经大致正确。否则你后面看到的问题，常常其实出在发射和运动阶段。

## 一条更适合新手理解的基础顺序

如果你现在要从零开始写第一篇帮助文档，并且希望后面能顺着主线一直扩展，建议先按这个顺序理解：

1. `nx 发射器（nxEmitter）`：先解决“粒子从哪里来”。
2. `nx 缓存（nxCache）`：再解决“结果怎么稳定回放”。
3. `nx 碰撞体（nxCollider）`：再解决“粒子会撞到什么”。
4. `nx 重力（nxGravity）`、`nx 风力（nxWind）`、`nx 湍流（nxTurbulence）`：再解决“粒子怎么动”。
5. `nx 流体（nxFluids）` 或 `nx 拖尾（nxTrail）`：最后决定“结果往哪种表现形式走”。
6. `nx 网格化（nxMesher）`：只在你需要可见表面网格时加入。

这个顺序的好处，是每一步都对应一个普通用户能直接观察的结果变化：

- 有没有出生
- 能不能稳定回看
- 会不会穿过去
- 运动有没有方向和扰动
- 最终是粒子、液体表面，还是线条轨迹

## 写第一篇修改器帮助文档时的落点建议

如果我们接下来就开始写第一篇，最适合先写的是 `nx 发射器（nxEmitter）`。  
原因很直接：它决定了后面几乎所有主线的起点，用户也最容易在这里建立“参数改动会带来什么结果”的第一层理解。

写 `nx 发射器（nxEmitter）` 时，建议同时把下面这几层浅联动一起写进去：

- 它和 `nx 缓存（nxCache）` 的关系：什么时候需要缓存来稳定观察
- 它和 `nx 碰撞体（nxCollider）` 的关系：为什么碰撞加入后，用户对发射是否正确的判断会变化
- 它和 `nx 流体（nxFluids）` 的关系：同样的发射设置进入流体分支后，用户更该先看流动是否合理
- 它和 `nx 拖尾（nxTrail）` 的关系：拖尾看到的问题，很多要先回头检查发射时序和运动路径

## 当前这条主线的定位

这不是完整的修改器总表，也不是唯一正确顺序。  
它更适合作为新版帮助文档的基础主线，用来决定：

- 哪些修改器属于主线必备角色
- 哪些修改器属于结果分支
- 哪些修改器更适合放在后续专题或扩展说明里

后面继续写单个修改器文档时，可以优先围绕这条主线展开。
